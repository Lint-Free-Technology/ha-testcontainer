# Generating Documentation Images and Animations

This guide explains how to use the `doc_image:` and `doc_animation:` keys in
scenario YAML files to automatically capture and verify documentation assets
(static screenshots and animated GIFs) as part of the test suite.

## Overview

Any scenario YAML that contains a `doc_image:` or `doc_animation:` key (or
both) participates in documentation-asset generation.  When
`test_doc_images.py` runs, it:

1. Pushes the scenario's card config to the test dashboard.
2. Runs the `setup:` block (if present) to prepare entity state.
3. Navigates to the view.
4. Runs the top-level `interactions:` block (if present).
5. Captures each `doc_image:` entry and each `doc_animation:` entry.
6. Compares captured assets to the stored files on disk.

If any asset differs beyond the configured `threshold`, the test fails — just
like a snapshot test.  Assets are regenerated by running with
`DOC_IMAGE_UPDATE=1`.

Pillow is required for animated GIF capture and for pixel-diff tolerance on
both images and animations.  Install it with:

```bash
pip install Pillow
```

## Where doc scenarios live

Scenarios can declare `doc_image:` / `doc_animation:` in **two locations**:

| Location | Purpose |
|---|---|
| `tests/visual/scenarios/*.yaml` | Regular test scenario that *also* produces a doc asset alongside its functional assertions |
| `docs/scenarios/*.yaml` | Documentation-asset-only scenario — no functional assertions; must declare `doc_image:` or `doc_animation:` |

Documentation-only scenarios must be placed under `docs/scenarios/` so that
`test_scenarios.py` (which runs functional assertions) does not try to execute
them.  `test_doc_images.py` loads scenarios from both locations.

Configure the `docs/scenarios/` path from your `conftest.py`:

```python
# tests/conftest.py
from pathlib import Path
import ha_testcontainer.visual.scenario_runner as sr

sr.SCENARIOS_DIR     = Path(__file__).parent / "scenarios"
sr.SNAPSHOTS_DIR     = Path(__file__).parent / "snapshots"
sr.REPO_ROOT         = Path(__file__).parent.parent
sr.DOCS_SCENARIOS_DIR = sr.REPO_ROOT / "docs" / "scenarios"
```

## Running the doc-image tests

```bash
# Run all doc-image / doc-animation tests
pytest tests/visual/test_doc_images.py

# Run a single scenario by id
pytest tests/visual/test_doc_images.py -k my_feature_card

# Regenerate ALL doc assets (overwrite existing files)
DOC_IMAGE_UPDATE=1 pytest tests/visual/test_doc_images.py

# Regenerate a single asset
DOC_IMAGE_UPDATE=1 pytest tests/visual/test_doc_images.py -k my_feature_card
```

Or via make (if your project defines these targets):

```bash
make test-visual              # run all visual tests including doc images
DOC_IMAGE_UPDATE=1 make test-visual   # regenerate all assets
```

## Static images — `doc_image:`

Add a `doc_image:` key to any scenario to capture a cropped screenshot and
write it to a file relative to the repository root.

### Minimal example

```yaml
id: my_feature_default
view_path: my-feature-test
card:
  type: my-card
  entity: light.bed_light

doc_image:
  output: docs/assets/my-feature-default.png
  root: my-card            # shadow-piercing selector — crop to this element
  padding: 8               # optional whitespace border in pixels (default 0)
  threshold: 0.02          # optional pixel-diff tolerance (default 0)
```

### Keys

| Key | Required | Description |
|---|---|---|
| `output` | ✅ | Path relative to the repository root where the PNG is written |
| `root` | | Shadow-piercing selector for the element to crop to. Omit for a full-viewport screenshot |
| `padding` | | Extra whitespace border — same CSS shorthand as `padding` (values in pixels, no units) |
| `threshold` | | Maximum fraction of pixels (0.0–1.0) allowed to differ between runs (default `0`) |
| `scale` | | Playwright screenshot scale: `"css"` (default) or `"device"` for HiDPI |
| `cursor` | | Render a cursor overlay: `"pointer"` / `"arrow"` / `none` (default) |
| `interactions` | | Additional interactions to run before *this* capture (list form only — see [Stepped captures](#stepped-captures)) |

### `padding` shorthand

Follows the same shorthand notation as CSS `padding`, but values are always
pixels (no units):

| Value | Meaning |
|---|---|
| `padding: 16` | 16 px on all sides |
| `padding: "16 8"` | 16 px top/bottom, 8 px left/right |
| `padding: "16 8 4"` | 16 px top, 8 px left/right, 4 px bottom |
| `padding: "20 8 8 8"` | 20 px top, 8 px right, 8 px bottom, 8 px left |

## Animated GIFs — `doc_animation:`

Add a `doc_animation:` key to capture a sequence of frames and assemble them
into an animated GIF.  **Pillow is required.**

### Minimal example (flat mode)

```yaml
id: my_feature_hover_anim
view_path: my-feature-test
card:
  type: my-card
  entity: light.bed_light

setup:
  - type: ha_service
    domain: light
    service: turn_on
    entity_id: light.bed_light

doc_animation:
  output: docs/assets/my-feature-hover.gif
  root: my-card
  padding: 8
  frames: 12
  interval_ms: 80
  threshold: 0.02
  cursor: pointer
  interactions:
    - type: hover
      root: my-card
      selector: ha-tile-icon
      settle_ms: 600
```

### Keys

| Key | Required | Description |
|---|---|---|
| `output` | ✅ | Path relative to the repository root where the GIF is written |
| `root` | | Shadow-piercing selector for the element to crop to. Omit for full viewport |
| `padding` | | Extra whitespace border (same shorthand as `doc_image`) |
| `frames` | | Total frames to capture in flat mode, or per-segment default (default `10`) |
| `interval_ms` | | Gap between frames in ms, also used as per-frame GIF display duration (default `100`) |
| `threshold` | | Maximum fraction of pixels per frame that may differ between runs (default `0`). Non-zero recommended (e.g. `0.02`) to absorb GIF palette quantisation differences |
| `scale` | | Screenshot scale: `"css"` (default) or `"device"` |
| `dither` | | Apply Floyd-Steinberg dithering when quantising to 256-colour GIF palette (default `true`). Set `false` only for flat-colour content |
| `cursor` | | Render a cursor overlay in every frame: `"pointer"` / `"arrow"` / `none` |
| `interactions` | | Interactions to run **before** the first frame (flat mode only). Same types as top-level `interactions:` |
| `segments` | | List of capture segments (segmented mode — see below). When present, top-level `frames:` and `interactions:` are ignored |

## Cursor and click-circle overlays

### Cursor overlay

Set `cursor:` to render a visible software cursor in the screenshot or
animation frame.  This is useful for documentation images where the mouse
position should be obvious to the reader.

```yaml
doc_image:
  output: docs/assets/hover-state.png
  root: my-card
  cursor: pointer    # pointing-hand — use when hovering over a clickable element
```

Accepted values:

| Value | Description |
|---|---|
| `"pointer"` / `"hand"` | Pointing-hand cursor |
| `"default"` / `"arrow"` | Standard arrow cursor |
| `none` (YAML null) / `"none"` | No cursor (default) |

In animations, `cursor:` at the top level applies to every frame.
Individual segments can override it (including setting `cursor: none` to hide
the cursor for that segment — see [Segmented animations](#segmented-animations)).

### Click-circle overlay (animations only)

In segmented animations, set `click_circle: true` on a segment to render a
semi-transparent filled circle centred on the last click position.  The circle
is automatically removed at the start of every other segment.

```yaml
doc_animation:
  output: docs/assets/toggle-click.gif
  root: my-card
  interval_ms: 80
  cursor: pointer
  segments:
    - interactions:
        - type: hover
          root: my-card
          selector: ha-tile-icon
          settle_ms: 400
      frames: 6
    - interactions:
        - type: click
          root: my-card
          selector: ha-tile-icon
          settle_ms: 800
      frames: 4
      click_circle: true   # circle shown while state change settles
    - frames: 8
      cursor: none         # circle auto-removed; cursor hidden after hover_away
```

## Stepped captures

`doc_image:` accepts either a **single mapping** or a **list of mappings** to
capture multiple states of the same scenario from one YAML file.  Each entry
can run its own `interactions:` before its screenshot is taken.

```yaml
doc_image:
  # State 1 — default (no interaction needed)
  - output: docs/assets/my-feature-default.png
    root: my-card
    padding: 8

  # State 2 — hovering
  - interactions:
      - type: hover
        root: my-card
        selector: ha-tile-icon
        settle_ms: 800
    output: docs/assets/my-feature-hover.png
    root: my-card
    padding: 8
    cursor: pointer

  # State 3 — clicked / active
  - interactions:
      - type: click
        root: my-card
        selector: ha-tile-icon
        settle_ms: 1500
    output: docs/assets/my-feature-active.png
    root: my-card
    padding: 8
```

> **Note** — interactions accumulate across entries.  The hover in entry 2 is
> still in effect when entry 3's click interaction runs.  Design your
> interaction sequences accordingly.

## Segmented animations

Use `segments:` to interleave state changes with frame capture, for example to
show an entity toggling on and off:

```yaml
doc_animation:
  output: docs/assets/toggle-animation.gif
  root: my-card
  padding: 8
  interval_ms: 100
  threshold: 0.02
  dither: true
  segments:
    # Segment 1 — entity off state
    - interactions:
        - type: ha_service
          domain: light
          service: turn_off
          entity_id: light.bed_light
          settle_ms: 400
      frames: 8

    # Segment 2 — entity on state
    - interactions:
        - type: ha_service
          domain: light
          service: turn_on
          entity_id: light.bed_light
          settle_ms: 400
      frames: 8
```

The crop area (`root` + `padding`) is **locked** to the element's dimensions
after the first segment's interactions settle.  All frames — including those
from subsequent segments — share the same pixel dimensions, even if the
element changes size between segments (e.g. a conditional card row appearing
or disappearing).

Each segment can also override the top-level `cursor:` key and declare
`click_circle: true` (see [Cursor and click-circle overlays](#cursor-and-click-circle-overlays)).

## Updating stored assets

The workflow mirrors snapshot tests:

| Scenario | Behaviour |
|---|---|
| Output file does not exist | Created automatically on the first run |
| `DOC_IMAGE_UPDATE=1` set | Existing file is always overwritten |
| Neither | Captured asset is compared to stored file; test fails if it differs beyond `threshold` |

After updating a component or its documentation images intentionally, run:

```bash
DOC_IMAGE_UPDATE=1 pytest tests/visual/test_doc_images.py
```

Then review and commit the regenerated PNG / GIF files.

## Complete annotated examples

### Single static image

```yaml
# docs/scenarios/tile_card_on.yaml
id: tile_card_on
description: "Tile card — light on state"
view_path: doc-tile-on

card:
  type: tile
  entity: light.bed_light

setup:
  - type: ha_service
    domain: light
    service: turn_on
    entity_id: light.bed_light
    data:
      brightness: 255

teardown:
  - type: ha_service
    domain: light
    service: turn_off
    entity_id: light.bed_light

doc_image:
  output: docs/assets/tile-card-on.png
  root: hui-tile-card
  padding: 8
  threshold: 0.01
```
